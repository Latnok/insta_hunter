# Продуктовая спецификация

## 1. Цель

Система должна помогать одному администратору находить Instagram-аккаунты, собирать данные профиля и последних рилсов, получать и оценивать transcript, запрашивать рекомендацию LLM и вручную принимать итоговое решение.

Первая версия предназначена только для поиска блогеров в нише одежды. Поддержка нескольких кампаний, клиентов или ниш не входит в MVP.

## 2. Пользователь и доступ

- Один администратор без ролей и управления пользователями.
- Логин задаётся через `ADMIN_USERNAME`.
- В конфигурации хранится только bcrypt-хэш пароля `ADMIN_PASSWORD_HASH`.
- Сессия серверная, хранится в PostgreSQL и действует семь дней с rolling expiration.
- Доступ к приложению только по HTTPS через Caddy.

## 3. Навигация

После входа доступны четыре основные вкладки:

1. **Кандидаты** — найденные или импортированные аккаунты до финального решения.
2. **Блогеры** — активные approved-аккаунты и фильтр архива.
3. **Рилсы** — собранные рилсы, transcript и качество текста.
4. **Очередь** — pipeline runs, jobs, попытки и ошибки.

Настройки открываются отдельной кнопкой с иконкой шестерёнки и не считаются пятой основной вкладкой.

## 4. Lifecycle аккаунта

Используется единая сущность Instagram-аккаунта. Кандидаты и блогеры — представления одного объекта с разным статусом.

Допустимые статусы и переходы:

```text
candidate -> approved -> archived -> approved
candidate -> rejected
```

Правила:

- `merge` отсутствует.
- Rating отсутствует.
- Hard delete через UI отсутствует.
- Rejected-кандидат остаётся доступен через фильтр и audit log; возврат rejected в candidate не входит в MVP.
- Архивировать можно только approved-блогера.
- Архив обратим: restore возвращает аккаунт в `approved`.
- Архивирование запрещает новые pipeline runs и отменяет ещё не начатые jobs аккаунта.

## 5. Добавление кандидатов

### 5.1 API-discovery

- Запускается только вручную.
- Администратор задаёт поисковый запрос и максимальное число результатов.
- Значение по умолчанию: 5 кандидатов.
- Допустимый диапазон: 1–100.
- Результаты сохраняются как candidates,

### 5.2 Ручной Instagram URL или username

- После валидации создаётся candidate.
- Для нового кандидата pipeline запускается автоматически.
- Повторный ввод не создаёт дубль и не сбрасывает существующий lifecycle-статус.

### 5.3 CSV

- Обязателен хотя бы один столбец: `url` или `username`.
- Допустимый дополнительный столбец: `source_note`.
- Legacy CSV вида «индекс, URL» без заголовков не поддерживается.
- Перед импортом UI показывает число валидных строк, дублей и ошибок и требует подтверждение.
- Для каждого нового кандидата после подтверждения автоматически создаётся pipeline.
- Массовых approve/reject или иных bulk-действий после импорта нет.

## 6. Кандидаты

Плитка кандидата показывает:

- avatar;
- username и ссылку на Instagram;
- display name и followers, если профиль доступен;
- состояние pipeline;
- число найденных и полезных рилсов;
- recommendation и confidence LLM;
- предупреждение об устаревших данных;
- быстрые действия `Обработать/Повторить`, `Approve`, `Reject`.

Правила решений:

- Reject доступен до и после завершения обработки.
- Approve заблокирован, пока нет валидного ответа LLM.
- Наличие валидного ответа разрешает человеку approve независимо от рекомендации модели.
- При неполных данных доступны только retry и reject.
- Любое решение требует подтверждения и записывается в audit log.

Полная информация, ошибки, сигналы LLM и force refresh открываются в боковой панели.

## 7. Блогеры

- Вкладка содержит только `approved`; `archived` показывается отдельным фильтром.
- На плитке доступны архивирование или восстановление.
- Ручной refresh находится в боковой панели.
- Refresh обновляет только профиль, рилсы и transcript и не запускает candidate evaluation.
- Автоматического периодического обновления нет.

## 8. Рилсы

Плитка рилса показывает:

- thumbnail;
- автора и дату публикации;
- play/like/comment counts, если они доступны;
- caption preview;
- transcript status, source и quality;
- ссылку для открытия оригинала в Instagram.

Видео не встраивается и не хранится постоянно.

## 9. Очередь

Пользователь видит:

- pipeline run и связанный аккаунт;
- job type и status;
- число попыток;
- время создания, старта и завершения;
- provider и краткую ошибку;
- подробности каждой попытки в боковой панели.

Доступные действия:

- retry для окончательно упавшей job;
- cancel для `pending`/`retry_wait` job;
- running job можно только просматривать.

Массовые действия отсутствуют.

## 10. Критерии и LLM

- В системе существует один активный набор критериев для одежды.
- Критерии и поисковые запросы хранятся в PostgreSQL с версионированием.
- Ручное редактирование всегда создаёт draft-версию.
- Draft активируется отдельным подтверждением; предыдущая active-версия становится `superseded`.
- Администратор может вручную запросить у LLM предложение новой версии.
- Для группового предложения используются только полноценно обработанные аккаунты, по которым человек уже принял решение `approved` или `rejected`.
- LLM создаёт draft и diff; автоматическая активация запрещена.
- Активация критериев не переоценивает старых кандидатов.
- Все запросы, сырые ответы и разобранные результаты LLM хранятся без автоматического удаления.

## 11. Интерфейс

- EJS + HTMX без React и `@mui/material`.
- Визуальный стиль повторяет Material Design собственными CSS-компонентами.
- Компактная адаптивная сетка: 3–4 колонки на desktop, одна колонка на узком экране.
- 24 плитки в первой порции; продолжение загружается кнопкой `Показать ещё`.
- Детали открываются в правой боковой панели без ухода со списка.
- По умолчанию новые/недавно изменённые сущности показываются первыми; рилсы сортируются по дате публикации.
- Предусмотрены поиск по username и фильтры по lifecycle, recommendation, job state, transcript state и freshness.
- Интерфейс двуязычный: русский и английский.
- При первом входе язык определяется по браузеру: `ru*` получает русский, остальные — английский.
- Ручной выбор языка сохраняется в cookie.
- В БД timestamps хранятся в UTC, в UI показываются в часовом поясе браузера.

## 12. Что не входит в MVP

- несколько администраторов и роли;
- несколько ниш или campaigns;
- публичный JSON API;
- экспорт CSV/JSON;
- массовые действия;
- автоматические расписания;
- автоматический approve;
- merge кандидатов;
- постоянное хранение видео или аудио;
